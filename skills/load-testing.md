# 負荷試験スターティングガイド

## 負荷試験とは

システムの性能を検証するためのテスト手法。以下を明らかにすることが目的：

- システムが想定しているキャパシティ（性能の許容値）を確保できているか
- 過剰な負荷によりどのようなシステム影響が発生するか
- 性能に関わるボトルネックがどこにあるか

## 4つの手法

負荷試験の目的（何をテストしたいか）に応じてテスト手法を選択する。

| 手法 | 概要 | 目的 |
|---|---|---|
| **ロードテスト**（Load Testing） | 一定水準のリクエストでテストする | 想定している性能を満たしているかどうかを検証する |
| **ストレステスト**（Stress Testing） | 限界を想定したリクエストでテストする | 性能限界の水準を明らかにし、許容値超過時のシステム障害を検証する |
| **ソークテスト**（Soak Testing）| 一定水準のリクエストを長時間かけてテストする | システムのリソース状況に応じてどのような影響が起きるか検証する |
| **ベースラインテスト** | 単一ユーザーのリクエストに基づいてテストする | 性能の基準値を特定し、他のテスト結果と比較してパフォーマンス低下を分析する |

## 負荷試験の流れ

### フェーズ1: 計画

#### 1. 目的設定

何のために負荷試験を実施するのかを明確化する。リスクとセットで考えると良い。

目的の例：
- 既存APIの改修により、APIの性能劣化がないことを確認したい
- 新機能を公開するが、想定トラフィックに耐えうるか検証したい
- APIのスロットリングやクォータを定めるために性能傾向を把握したい

#### 2. 調査

目的に応じて、試験対象の構成・仕様・アクセス状況などを調査する。

- インフラ構成
- 機能の仕様
- システムリソース利用状況
- アクセス状況
- 試験実施環境の確認

> 準備が不足すると試験実施時に場当たり的な対応が増え、試験結果の信憑性に影響を与える。

#### 3. 性能目標と計測指標の設定

目的の達成基準として性能目標を具体的に定める。性能目標が具体的であればあるほど、計測指標も想定しやすくなる。

例：
- 同時接続1,000人以上でトップ画面にリクエストがあるとき、レスポンスタイムが平均800ms以下であること
- 定常時のリクエスト量を営業日を想定した期間継続的に受け付けたとき、CPU使用率が平均20%前後で推移すること

#### 4. テストシナリオ作成

試験対象がユーザーにどのように使われるかに基づいてシナリオを作成する。

- ユーザーがどのような状態を持ってリクエストするかを明確にする
- エミュレートする必要がある処理を書き起こす

例（マイページのレスポンスタイム検証）：
1. ログインページにアクセスする
2. ログインボタンを押下する（認証情報がCookieにセット、ユーザー情報取得APIが呼び出される）
3. マイページに遷移する（認証処理を経てリダイレクト、設定情報取得APIが呼び出される）

### フェーズ2: 実施

#### 1. 準備

- **実施環境の準備**: 負荷試験ツール・インフラ環境の整備
- **テストデータの準備**
- **関係者への共有**: 負荷試験の影響を社内外の関係者に事前共有し、影響範囲を見積もっておく
- **監視対象の確認**: 利用するログ・メトリクスを整理し、監視ダッシュボードを事前に用意しておく

#### 2. テスト実施

シナリオに従ってテストを実施する。

#### 3. 結果分析

以下の観点で観察し、原因や対策を検討する：

- スパイクが発生している箇所
- 増加・低下傾向のあるメトリクス
- 見慣れないエラーの頻発

#### 4. チューニング

分析結果に基づいて必要なチューニングを行い、必要であれば再実施する。

## 評価のための問い

- 負荷試験の目的は明確か？目的がないと後続のプロセスが意味を持たなくなる
- 4つの手法（ロード/ストレス/ソーク/ベースライン）のうち、目的に合った手法を選んでいるか？
- 性能目標に具体的な数値（閾値）が定められているか？
- テストシナリオはユーザーの実際の使われ方を反映しているか？
- 試験実施前に関係者への共有と影響範囲の見積もりができているか？
- 監視ダッシュボードを事前に用意できているか？
- チューニング後に再試験を実施しているか？
