# Architecture Strategy Evaluation

良い戦略と悪い戦略を見分けるための評価基準。

## 良い戦略の3要素

良い戦略は必ず以下の3要素を含む:

1. **課題分析（Diagnosis）**:
   - 現状の問題を明確にする
   - 根本原因と表面症状を区別する
   - 複数の課題がある場合、優先順位をつける

2. **基本方針（Guiding Policy）**:
   - 問題へのアプローチ方向を示す
   - Why（なぜその方針か）を明確にする
   - トレードオフを明示する

3. **施策の方向性（Coherent Actions）**:
   - 実施内容の大枠を定義する
   - 方針を実現するための具体的なアクション
   - 優先順位と段階的な計画

## 悪い戦略の3つの特徴

以下の特徴がある場合、悪い戦略である可能性が高い:

1. **抽象性が高い（スローガンだけ）**:
   - ✗ 悪い例: 「クラウドネイティブなシステムを構築する」
   - 何をすべきかが見えない
   - 施策の内容が不明確

2. **本質回避（表面的な対処）**:
   - ✗ 悪い例: 「パフォーマンスが悪い」→「サーバーを増強する」
   - 根本原因を分析していない
   - 一時しのぎの対処に留まる

3. **目標と戦略の混同**:
   - ✗ 悪い例: 「デリバリー速度を2倍にする」（これは目標）
   - 達成状態は書かれているが、道筋がない
   - どうやって実現するかが示されていない

## Before / After の例

**Before（悪い戦略）**:
```
【戦略】
マイクロサービス化を推進する

【理由】
スケーラビリティと保守性を向上させるため

【施策】
- フロントエンドとバックエンドを分離
- サービスごとにチームを編成
```

問題点:
- 課題分析が不足（なぜマイクロサービス化が必要か不明）
- 手段（マイクロサービス化）が目的化している
- 現状の何が問題で、それがどう解決されるのか不明

**After（良い戦略）**:
```
【課題分析】
現状: モノリス構成により、デプロイが相互依存し、リリース速度が低下
- 問題: 一部の変更で全体をデプロイする必要があり、チーム間調整が頻発
- 影響: 機能リリースまで平均2週間かかり、市場投入速度が遅い
- 根本原因: すべての機能が単一アプリケーションに含まれている

【基本方針】
Why: チーム自律性を高め、機能ごとに独立してリリースできるようにする
What: 機能ドメインごとに境界を設け、独立したデプロイを可能にする

【施策の方向性】
1. 機能ドメインの分割基準を策定（ビジネスケイパビリティ単位）
2. 段階的な移行計画（ユーザー管理→注文管理→...の順）
3. 各ドメインのインターフェースを明確化
```

## よくある失敗パターン

1. **手段から入る**:
   - ✗ 「マイクロサービス化する」
   - ◯ 「なぜマイクロサービス化が必要か」から始める

2. **課題分析が浅い**:
   - ✗ 「パフォーマンスが悪い」→ 終わり
   - ◯ なぜ悪いのか、どこがボトルネックか、ビジネスへの影響は？

3. **施策が抽象的**:
   - ✗ 「技術的負債を削減する」
   - ◯ どの負債を、どの順番で、どう削減するか

4. **トレードオフを明示しない**:
   - ✗ 「スピードと品質を両立する」
   - ◯ 現フェーズではスピードを優先、品質は〇〇で担保
