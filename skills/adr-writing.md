# ADR Writing: 良いADRを書くための4原則

ADR（Architecture Decision Record）を書く際に、内容が形骸化・バラつかないための判断基準。

## 1. 意思決定を一つに絞る（Atomicity）

**1つのADRには1つの意思決定だけを書く。**

複数トピックを1つにまとめると:
- 将来の変更時に「全体を無効化するか、部分修正するか」の管理が複雑になる
- レビュー時に「Aには賛成だがBには反対」という状況で合意形成がストップする

**判定サイン**: タイトルに「A と B について」と書きたくなったら、2つのADRに分割するタイミング。

## 2. 「なぜ（Why）」を記録する（Decision Point）

**ADRはシステムの説明書ではなく、分岐点（意思決定ポイント）の記録。**

コードを見れば「どう動くか（How）」はわかる。ADRに残すべきはコードから読み取れないもの:
- **選択の理由**: なぜその選択肢を採用したか
- **捨てた選択肢**: 検討したが採用しなかった案とその理由

「結果（Consequences）」セクションはメリットだけでなく、**トレードオフとリスクを必ず記録する**。

> 「開発スピードを優先するために、整合性の担保を一部犠牲にする」といった判断こそが将来にわたって価値ある情報になる。

## 3. 議論を経てから書く

**ADRはチームの合意を固定化する「決定の証」であり、下書きではない。**

推奨プロセス:
1. ホワイトボードや軽量ドキュメントでチームと議論し、合意形成を図る
2. 書くべき事項を箇条書きで書き出し、アウトラインベースでレビューを始める
3. 合意が得られたら ADR として清書する

議論の発散→収束のプロセスを経ることで、チーム全体の理解と納得感が高まり、ADR 自体の品質も向上する。

## 4. テクニカルライティングを心がける

**誰が書いても文章の質が一定になるような記述を意識する。**

テクニカルライティングの原則（参考: [Google Technical Writing](https://developers.google.com/tech-writing)):
- 一文一義（1文に1つの情報）
- 能動態を使う
- 曖昧な言葉（「適切に」「うまく」）を避け、具体的な基準で書く
- 事実と意見を区別する

## フォーマット選択時の注意

ADRのフォーマットは自由度が高く、特に「結果（Consequences）」セクションは何を書くか迷いやすい。

カスタマイズ時は **トレードオフを独立したセクションとして設ける** ことで、意思決定の核心が形骸化することを防げる。

## 良いADR・悪いADRの見分け方

| 観点 | 良いADR | 悪いADR |
|------|--------|--------|
| スコープ | 意思決定が1つ | 複数の決定が混在している |
| Why | 選択の理由とトレードオフが明記されている | メリットだけ列挙（デメリット・リスクがない） |
| 合意 | チームで議論・合意済み | 個人の判断をそのまま記録 |
| 文体 | 簡潔・具体的・事実ベース | 曖昧・冗長・主観が混在 |
| 捨てた選択肢 | 記録されている | 採用した選択肢しか書かれていない |
