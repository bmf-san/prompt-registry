# アーキテクチャドキュメントを書くときに気をつけること

## 概要

アーキテクチャドキュメントは、関係者に対して設計の妥当性を説明し、合意を形成するための資料として機能する。
単なる設計情報の羅列ではなく、読者の理解と納得を導くための戦略的な文書であるべき。

## 目的を明確にする

ドキュメント作成で最初に意識すべきは目的。

- この文書が**誰に**向けたものか
- 何を**伝えること**を目的としているか

主な対象はプロジェクトのステークホルダー。ステークホルダーが誰かがドキュメントの内容を左右する変数になる。
技術的な背景を持たないステークホルダー（プロジェクトマネージャー・セールスなど）を対象とする場合は、より注意深い配慮が必要。

## ドキュメントが備えるべき6つの特性

| 特性 | 説明 | 実践のポイント |
|---|---|---|
| **正確性** | 対象となるアーキテクチャとステークホルダーの関心事に対して正確である | 提案されたアーキテクチャがステークホルダーのニーズに合致していることを明確にする |
| **十分性** | なぜその構成を採用したかという判断理由を示す | 代替案がある場合はその比較・却下理由も論理的に説明し、説得力を高める |
| **簡潔性** | 重要な設計判断に焦点を絞る | ステークホルダーの技術力・問題の複雑さ・利用可能なリソースに応じて詳細度を調整する |
| **明瞭性** | 読み手の知識・理解の程度に合わせた表現にする | 専門用語の使い分けや、図・表による視覚的な伝え方を工夫する |
| **通用性** | 最新の設計状況を反映している | 設計の進化に合わせてドキュメントを定期的に更新する |
| **精密性** | 実装に着手できる程度に必要な詳細を記述する | 詳細に書きすぎると可読性が損なわれるため、図・別文書・具体と抽象の使い分けでバランスを取る |

> 著者は**簡潔性を最初に意識**し、その後他の観点を追加していくプロセスを取る。最初から詳細を含めすぎると認知負荷が高くなるため、まず全体像を把握してから必要な情報を追加する形でドキュメントを構築する。

## 推奨する構成

| セクション | 内容 |
|---|---|
| **概論** | ドキュメントの目的・想定読者・システム目標の要約・スコープ・解決策の要約 |
| **要件定義** | 機能要件・非機能要件・特記事項（規制・前提条件・制約など） |
| **アーキテクチャ図** | 構成要素とその関係・外部システムとの関係（C4モデルが有効）・信頼境界・可用性の観点 |
| **データモデル** | 主要なデータ構造の図（ER図・スキーマ図など）・個人情報の取り扱い・法的制約 |
| **ユースケース** | 主要な利用シナリオ、設計判断の背景となる業務文脈 |
| **技術詳細** | 技術スタック・フレームワーク・ライブラリ、技術選定理由・設計上の工夫 |
| **リスクと課題** | 想定されるリスク、技術的・業務的な課題と対応方針 |
| **展望** | 将来的な拡張・改善の方向性、可用性・スケーラビリティ・保守性の展望 |
| **未解決事項** | 今後の検討が必要なポイント・保留中の課題 |
| **関連ドキュメント** | 詳細仕様書・業務フロー図・プロジェクト計画書などへのリンク |

## 評価のための問い

- ドキュメントの想定読者は誰か、明確に定義されているか？
- 設計の判断理由（なぜその構成を選んだか、代替案との比較）が記述されているか？
- 簡潔性を意識して重要な設計判断に焦点が絞られているか？
- 専門用語が読者の技術レベルに合わせて適切に使われているか？
- 設計の進化に合わせてドキュメントが最新の状態に保たれているか？
- 複数のことを一度に説明しようとしていないか（情報を整理し焦点を絞れているか）？
- 図や別文書を併用して具体と抽象のバランスが取れているか？
