# データベースインデックス評価スキル

## インデックスとは

テーブルに格納されているレコードを**高速に取り出すための仕組み**。

```sql
-- O(n)の問題を持つクエリ
SELECT * FROM users WHERE first_name = 'Tom';

-- インデックスを追加して高速化
ALTER TABLE users ADD INDEX (first_name);
```

---

## メリット・デメリット

| 観点 | 内容 |
|------|------|
| **メリット** | データの読み込み・取得の速度向上 |
| **デメリット（容量）** | インデックス分の記憶領域が増加 |
| **デメリット（書き込み速度）** | データの作成・更新時にインデックスの追加・更新も同時に行われる |

---

## インデックスのパターン

| パターン | 構文例 | 用途・特徴 |
|---------|-------|---------|
| **通常**（1カラム） | `ADD INDEX (first_name)` | 基本形 |
| **部分的インデックス** | `ADD INDEX (first_name(4))` | 先頭Nバイトのみ。容量増加を抑えつつ性能向上 |
| **マルチカラムインデックス**（複合・合成） | `ADD INDEX (last_name, first_name)` | 1クエリにつき1インデックスの制約を複数カラムでカバー。先頭カラムはカーディナリティの高いものを選ぶ |
| **ユニークインデックス** | `ADD UNIQUE (first_name)` | NULL以外の値の重複を禁止。MySQLではユニークキー指定で自動付与 |

---

## インデックスの効果測定

`EXPLAIN` でクエリの実行計画を確認する。

```sql
EXPLAIN SELECT * FROM users WHERE first_name = 'Tom';
```

確認すべき項目:

| 項目 | 内容 |
|------|------|
| `possible_keys` | 選択可能なインデックス |
| `key` | 実際に選択されたインデックス |
| `extra: Using filesort` | ソートに必要なメモリが不足し物理ファイルに書き出してソート → 最適化を検討 |
| `extra: Using temporary` | クエリ実行のためにテンポラリテーブルが作成されている → 最適化を検討 |

---

## インデックスを貼る基準

以下はあくまで**目安**。必ずEXPLAINで計測すること。

- テーブルのデータ量が多く、検索対象レコードが少量になる時
- `WHERE`、`JOIN`、`ORDER BY` などに利用されるカラムがある場合
- NULLを多く含むデータからNULL以外の検索をする（インデックスにNULLが含まれないため有効な場合がある）
- データの追加・更新・削除が頻繁でないデータ（インデックスの更新負荷を考慮）

---

## クラスタインデックスとセカンダリインデックス

| 種別 | 定義 | 特記事項 |
|------|------|---------|
| **クラスタインデックス** | 主キーで定義されたカラム。NOT NULLのユニークキー。該当なければInnoDBが`GEN_CLUST_INDEX`（非表示）を自動生成 | テーブルデータの物理的な並び順と一致 |
| **セカンダリインデックス** | クラスタインデックス以外のすべてのインデックス | 主キーの値を含む → カバリングインデックス設計時に主キーを複合インデックスに含めなくても良い場合がある |

---

## カバリングインデックス

クエリの実行結果に必要なすべてのカラムを含むインデックス。

- データファイルを読まずインデックスだけでカバーできるため検索が高速化
- セカンダリインデックスは主キーを含むため、意識せずにカバリングインデックスになるケースがある

---

## インデックスが効かないパターン（注意）

| パターン | 例 | 理由 |
|---------|-----|------|
| **インデックス列の演算・SQL関数** | `WHERE amount * 2 > 10` | インデックスに演算結果は保持されていない（`WHERE amount > 5` に書き換える） |
| **IS NULL / IS NOT NULL** | `WHERE amount IS NULL` | 基本的にインデックスが有効活用されない（DBMSの仕様に依る） |
| **否定形・OR** | `WHERE amount <> 10` | 否定形はインデックスを活用できない |
| **LIKE（前方一致以外）** | `WHERE name LIKE '%abc'` | B-Treeの性質により前方一致のみ有効（`'a%'` は有効） |
| **暗黙的な型変換** | `WHERE age = '10'`（ageが数値型） | 暗黙的な型変換が発生するとインデックスが無効化される |

---

## 評価に使える問い

- `EXPLAIN` で実際のインデックス使用状況（`key`、`Extra`）を確認しているか？
- `WHERE`・`JOIN`・`ORDER BY` に使用するカラムにインデックスが設定されているか？
- マルチカラムインデックスの先頭カラムはカーディナリティの高いものが選ばれているか？
- インデックス列に演算・関数・暗黙的型変換が適用されていないか？
- LIKE検索で後方一致（`'%abc'`）を使っていないか？
- 書き込み頻度の高いテーブルで不要なインデックスが乱立していないか？
- カバリングインデックスの活用でデータファイルへのアクセスを削減できるか？
