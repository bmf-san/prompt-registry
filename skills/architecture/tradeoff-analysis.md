## トレードオフ分析の基本原則

技術選定や設計判断では、必ず複数の選択肢が存在し、どれも完璧ではない。目標は「最良の選択」ではなく「この文脈での最適な選択とその根拠の明示」。

**LLM が陥りやすい失敗パターン（これを避ける）:**

| 失敗パターン | 説明 |
|------------|------|
| 教科書的回答 | 文脈を無視して「一般的にはAが良い」と答える |
| 片側のみ提示 | メリットか デメリットの一方だけを強調する |
| チーム制約の無視 | チームのスキルセット・規模・文化を考慮しない |
| 結論の曖昧化 | 「どちらも良い」で結論を避ける |

---

## 技術選定の評価基準

| 観点 | 評価項目 |
|------|---------|
| **要件適合性** | 機能要件・非機能要件を満たすか |
| **成熟度** | 本番環境での実績、コミュニティの活発さ |
| **学習コスト** | チームが習得できるか、既存スキルとの距離 |
| **エコシステム** | ライブラリ、ツール、サポートの充実度 |
| **ライセンス** | 商用利用可能か、ライセンスの制約 |
| **長期保守性** | 将来的にサポートされるか、廃止リスク |
| **コスト** | ライセンス費用、インフラコスト、運用コスト |
| **可逆性** | 選択を変更する際のコストはどの程度か |

## 技術比較フォーマット

```
【技術A vs 技術B vs 技術C】

| 評価項目 | 技術A | 技術B | 技術C |
|---------|-------|-------|-------|
| 要件適合性 | ◎ | ○ | △ |
| 学習コスト | △ | ○ | ◎ |
| エコシステム | ◎ | ○ | △ |
| 長期保守性 | ◎ | ◎ | △ |
| コスト | △ | ○ | ◎ |

◎: 非常に良い  ○: 良い  △: 課題あり  ✗: 不適

【推奨】
技術A を推奨します。

【理由】
- <要件との対応>
- <チームのスキルセットとの整合>
- <長期的なリスク評価>

【前提条件】
- この推奨は<特定の制約>を前提にしています
- <条件>が変わる場合は技術B を検討してください

【代替案】
技術B も選択肢として検討可能です（<どのような場合に有効か>）
```

## Build vs Buy の判断フレームワーク

社内開発（Build）か外部サービス利用（Buy）かの判断指標:

| 判断軸 | Build を選ぶ | Buy を選ぶ |
|-------|------------|---------|
| **差別化への寄与** | 競争優位の核心 | 汎用的な機能 |
| **チームの専門性** | 内製できる技術力がある | 専門チームが不要 |
| **コスト（TCO）** | 長期的に内製が安い | 外部サービスが安い |
| **時間** | 開発期間が許容できる | 今すぐ必要 |
| **コンプライアンス** | データを外部に出せない | 問題なし |
| **ベンダーリスク** | 依存を避けたい | 許容できる |

## 設計判断の提示フォーマット

```
【現在の設計】
<問題のある設計または選択肢Aの説明>

【問題点・トレードオフ】
- <問題・懸念点>
- <何を犠牲にするか>

【改善案・推奨】
<推奨する設計または選択肢>

【改善理由】
- なぜこの判断に至ったか
- どのような利点があるか
- どのようなトレードオフが残るか

【影響範囲・前提条件】
- この判断の有効条件
- 変更が必要になるトリガー
```

## チェックリスト

- [ ] 複数の選択肢を比較したか（1つだけ提示していないか）
- [ ] チームのスキルセット・規模・文化を考慮したか
- [ ] コスト（TCO: 総所有コスト）を比較したか
- [ ] 可逆性（後で変更できるか）を評価したか
- [ ] 推奨に明確な根拠を明示したか
- [ ] 前提条件（この推奨が有効な条件）を明記したか
- [ ] 教科書的な回答になっていないか（文脈を反映しているか）
