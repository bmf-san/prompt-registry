# DDD（ドメイン駆動設計）スキル

ドメイン駆動設計（Domain-Driven Design）の主要概念と実践的な適用方法を定義するスキル。

## 使用するエージェント

- [app-architect](../../agents/app-architect.md)

---

## 主要概念

### エンティティ（Entity）

一意の識別子を持つオブジェクト。属性が変わっても同一性が保たれる。

```
User（ユーザー）
  識別子: userId
  属性: name, email
  振る舞い: register(), updateProfile()
```

### 値オブジェクト（Value Object）

属性の組み合わせで識別されるオブジェクト。不変（immutable）であること。

```
Email（メールアドレス）
  属性: value
  制約: メールアドレス形式の検証
  ※ 同じ値を持つ2つの Email は等しい
```

### 集約（Aggregate）

一貫性境界を持つオブジェクトのまとまり。トランザクション整合性を保証する単位。

- **集約ルート（Aggregate Root）**: 集約への唯一のエントリーポイント
- 外部から集約内の要素を直接操作しない
- 集約をまたがる整合性は「結果整合性」で対処する

```
Order（注文）集約
  集約ルート: Order
  エンティティ: OrderItem
  値オブジェクト: Money, Quantity
  ビジネスルール: 注文は最低1つの注文アイテムを持つ
```

### ユビキタス言語（Ubiquitous Language）

ドメインエキスパートと開発者が共通で使う言語。コードの命名に反映させる。

- ✗ `processOrder()` → ◯ `confirm()`, `ship()`, `cancel()`
- ✗ `updateUserData()` → ◯ `changeEmail()`, `promoteToAdmin()`

### 境界づけられたコンテキスト（Bounded Context）

同じモデルが通用する境界。コンテキストをまたぐ場合はモデルの変換が必要。

**コンテキスト間の関係パターン:**

| パターン | 説明 | 適用場面 |
|---------|------|---------|
| 共有カーネル | 共通モデルを両チームで共有 | 密結合が許容できる場合 |
| 順応者（Conformist） | 下流が上流のモデルに従う | 上流チームに交渉力がない場合 |
| 腐敗防止層（ACL） | 下流がアダプターで変換する | モデルの独立性を保ちたい場合 |
| 公開ホスト | 上流がAPIを公開する | 複数の下流がいる場合 |

## ドメインモデル設計のフォーマット

```
【エンティティ】
- <名前>（<日本語名>）
  - 識別子: <id フィールド>
  - 属性: <フィールド一覧>
  - 振る舞い: <メソッド一覧>

【値オブジェクト】
- <名前>（<日本語名>）
  - 属性: <フィールド一覧>
  - 制約: <バリデーション>

【集約】
- <名前>（<日本語名>）集約
  - 集約ルート: <ルートエンティティ>
  - エンティティ: <内部エンティティ一覧>
  - 値オブジェクト: <VO一覧>
  - ビジネスルール: <整合性ルール>
```

## 適用の判断基準

### DDD を適用すべき場面

- ビジネスロジックが複雑で、ドメインエキスパートとの継続的な協働が必要
- 同じ概念が複数のコンテキストで異なる意味を持つ
- 長期的な保守・拡張性が重要

### DDD を適用しない（過剰設計になる）場面

- CRUD 中心のシンプルなデータ管理システム
- ドメインロジックがほぼない汎用サブドメイン（ログ管理、メール送信など）
- 短期プロジェクトで将来的な拡張が想定されない

## チェックリスト

- [ ] エンティティと値オブジェクトを適切に区別しているか
- [ ] 集約の境界は整合性要件に基づいて設定されているか
- [ ] 集約が不必要に大きくなっていないか
- [ ] ユビキタス言語がコードに反映されているか
- [ ] Bounded Context の境界は明確か
- [ ] コンテキスト間の統合方式は明示されているか
- [ ] 過剰設計になっていないか（汎用サブドメインへの DDD 乱用NG）
