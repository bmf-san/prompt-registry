# Webアプリケーションの処理モデル評価スキル

## 3つの処理モデル概要

| モデル | 特徴 | 代表例 |
|--------|------|-------|
| **イベントループ**（非同期・シングルスレッド） | 1つのスレッドでイベント（非同期IOなど）を順に処理 | Node.js、Deno、JavaScript（ブラウザ） |
| **スレッド**（マルチスレッド） | 1プロセスの中で複数のスレッドが並行に動作 | Java、Python、Go |
| **プロセス**（マルチプロセス） | 各リクエストを別プロセスで処理。完全分離 | PHP（FPM）、Ruby（複数Pumaプロセス）、Sidekiq |

---

## 各モデルのメリット・デメリット

### イベントループモデル

```
[ イベントキュー ] → [ イベントループ ] → [ 実行 ]
```

| メリット | デメリット |
|---------|-----------|
| メモリ効率が良い（スレッドやプロセスを大量に作らない） | 重い計算（CPUバウンド）でブロックしやすい |
| IO待ちを他の処理に活用できる | 単一スレッドのため真の並列処理は不可能 |

### スレッドモデル

```
[ プロセス ]
 ├─ スレッド1（リクエストA）
 ├─ スレッド2（リクエストB）
 └─ ...
```

| メリット | デメリット |
|---------|-----------|
| CPUバウンド処理に強い（マルチコア活用可能） | スレッドごとのリソース消費や競合状態の管理が必要 |

### プロセスモデル

```
[ リクエストA ] → [ プロセス1 ]
[ リクエストB ] → [ プロセス2 ]
```

| メリット | デメリット |
|---------|-----------|
| 安定性が高い（他の処理の影響を受けにくい） | メモリ効率が悪い（プロセスごとに環境を持つ） |

## 言語ごとの処理モデルと向き・不向き

| 言語・ランタイム | 処理モデル | IOバウンド | CPUバウンド | 特記事項 |
|----------------|-----------|-----------|------------|---------|
| Node.js | イベントループ（非同期） | ◎ | △ | IO特化、重い計算処理は注意 |
| Go | goroutine + M:Nスレッド | ◎ | ◎ | 軽量スレッドで並列・並行処理が得意 |
| Ruby（MRI） | スレッド＋GIL（制限あり） | ○ | △ | GILによりCPU処理は実質シングルスレッド |
| PHP（FPM） | プロセス（1リクエスト=1プロセス） | ○ | △ | プロセス分離で安定だがオーバーヘッド大 |
| Java | マルチスレッド | ◎ | ◎ | JVMレベルでのスレッド最適化 |
| Python（CPython） | スレッド＋GIL（制限あり） | ○ | △ | Ruby MRIと同様にGILの制約あり |

## 実装上の注意点

### イベントループモデル
- **単一スレッド**: 真の並列処理は不可能
- **ノンブロッキングIO**: 同期処理はイベントループをブロックする
- **CPUバウンド回避**: 重い計算はWorker Threadsや外部プロセスに委譲

### スレッドモデル
- **競合状態**: 共有メモリへの同時アクセス制御が必要
- **デッドロック**: 複数ロックの取得順序に注意
- **メモリオーバーヘッド**: スレッドあたり約1〜8MBのスタック領域

### プロセスモデル
- **プロセス間通信**: IPCやファイルシステム経由での連携
- **メモリ使用量**: 各プロセスが独立したメモリ空間を持つ
- **起動コスト**: プロセス作成はスレッド作成より重い

---

## 選択指針

- **IOバウンドが多い**: イベントループモデルが適している
- **CPUバウンドが多い**: スレッドモデルやプロセスモデルが適している
- **安定性・分離を優先**: プロセスモデルが適している
- 言語ごとの特性（GILの有無など）も考慮する

---

## 評価に使える問い

- アプリケーションのワークロードはIOバウンドかCPUバウンドか？
- 使用言語・ランタイムの処理モデルと、ワークロードの特性が合致しているか？
- イベントループモデルを使う場合、CPUバウンドな処理が同一スレッドをブロックしないか？
- スレッドモデルを使う場合、共有資源へのアクセス制御（ロック・競合状態）が設計されているか？
- GILが存在する言語（Ruby MRI、Python CPython）でCPU並列化を期待した設計になっていないか？
- プロセスモデルを使う場合、プロセス間通信のオーバーヘッドは許容範囲か？
