# Writing Guide: プロンプト作成ガイド

このリポジトリにプロンプトを追加・編集する際の規約です。

## プロンプトの基本構造

プロンプトは以下のコンポーネントで構成されます。すべてを含める必要はなく、タスクに応じて取捨選択します。

| コンポーネント | 内容 | 例 |
|---|---|---|
| **Role** | AIに与える役割・ペルソナ | 「あなたはシニアのソフトウェアエンジニアです」 |
| **Task** | 実行してほしい具体的なタスク | 「以下のコードのバグを特定し、修正してください」 |
| **Input** | 処理対象のデータや文章 | `<input>ユーザーから受け取ったテキスト</input>` |
| **Output Format** | 出力の形式・構造 | 「JSON形式で返してください」「箇条書き3点以内で」 |
| **Guidelines** | 守るべきルールや方針 | 「専門用語は使わず、中学生でもわかる言葉で書く」 |
| **Prohibited Actions** | やってはいけないこと | 「個人情報を含む内容は出力しないでください」 |
| **Timestamp / User Info** | 実行時刻やユーザー属性などのメタ情報 | 「現在日時：2026-03-03 / ユーザー権限：管理者」 |
| **Knowledge Base** | 参照させたい外部ドキュメント・背景知識 | FAQ文書、社内規定、検索結果など |
| **Example** | 期待する出力の具体例（Few-shot） | 入力と出力のペアを1〜数件 |

## プロンプト作成の全体指針

| 指針 | 説明 |
|---|---|
| **曖昧語を排除する** | 「うまく」「適切に」など主観的な言葉を避け、「3行以内で」「箇条書きで」など客観的な基準に置き換える |
| **否定形を避ける** | 「〜しないで」より「〜してください」と肯定形で書く方がモデルは従いやすい |
| **指示を矛盾させない** | 「簡潔に、かつ詳しく説明して」のような矛盾した制約は出力を不安定にする |
| **タスクは1つに絞る** | 1つのプロンプトで多くを求めると質が下がる。複数タスクは分割する |
| **指示・質問は末尾に書く** | 「Lost in the Middle」現象により中間の情報は忘却されやすい。参照データは冒頭、具体的な指示や質問は末尾に配置する |
| **how より what を重視する** | 「どうやるか」を細かく指示するより「何を出力してほしいか」を明確にする方が柔軟性が高い |

## プロンプト設計フレームワーク

プロンプト全体を構造化するためのテンプレート。基本パターンを組み合わせる際の骨格として使います。

| フレームワーク | 構成要素 | 向いているケース |
|---|---|---|
| **COSTAR** | Context / Objective / Style / Tone / Audience / Response | 汎用。出力の制御性を高めたいとき |
| **CREATE** | Character / Request / Examples / Adjustment / Type / Extras | 専門家ペルソナを使った分析・創造的タスク |
| **KERNEL** | Keep it simple / Easy to verify / Reproducible / Narrow scope / Explicit constraints / Logical structure | トークン削減と精度を両立したいとき |
