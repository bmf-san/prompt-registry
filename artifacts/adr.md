# Artifact: Architecture Decision Record (ADR)

アーキテクチャ上の意思決定を記録するためのドキュメントテンプレート。
技術的な判断とその根拠を残すことで、将来のチームメンバーが決定の背景を理解できるようにする。

## テンプレート

```markdown
# ADR-{連番}: {決定のタイトル}

- **日付**: YYYY-MM-DD
- **ステータス**: Proposed | Accepted | Deprecated | Superseded by ADR-{N}
- **決定者**: {名前またはチーム}

## コンテキスト

この決定が必要になった背景・状況を説明する。
解決しようとしている問題、関係するシステム、制約条件を記述する。

## 検討した選択肢

### 選択肢 A: {名前}

**概要**: {1〜2文で説明}

**利点**:
- {利点1}
- {利点2}

**欠点**:
- {欠点1}
- {欠点2}

### 選択肢 B: {名前}

**概要**: {1〜2文で説明}

**利点**:
- {利点1}

**欠点**:
- {欠点1}

## 決定

**選択した選択肢**: {選択肢の名前}

### 理由

- {選択した主な理由}
- {根拠となる要件や制約}

### トレードオフの受け入れ

- {受け入れる欠点とその緩和策}

## 結果

この決定により期待される影響を記述する。

**肯定的な影響**:
- {影響1}

**否定的な影響・リスク**:
- {リスク1}（対応策: {対応策}）

## 関連するADR・参考資料

- [ADR-{N}: {関連するADRのタイトル}](./adr-{N}-{slug}.md)
- {参考資料URL}
```

## 書き方のガイドライン

### ステータスの使い分け

| ステータス | 意味 |
|-----------|------|
| `Proposed` | 提案中。まだ合意が取れていない |
| `Accepted` | 採用決定済み |
| `Deprecated` | 理由は不明だが無効になった |
| `Superseded by ADR-N` | より新しい ADR-N に置き換えられた |

### タイトルの書き方

動詞始まり・能動態で書く:
- ✅ 「認証方式にJWTを採用する」
- ✅ 「フロントエンドの状態管理にZustandを使用する」
- ❌ 「JWT について」
- ❌ 「認証の検討」

### コンテキストに書くこと

- 問題が生じた背景（ビジネス要件、技術的な制約）
- この決定をしなければならない理由
- 知っておくべき現状の状況

### 「決定した理由」に書くこと

- 採用した選択肢の根拠を具体的に書く
- 「〇〇という要件があるため」「チームが△△に慣れているため」など、後から見て納得できる理由を記載する
- 感情的・曖昧な理由（「シンプルだから」だけ）は避ける

## ファイル命名規則

```
adr-{4桁連番}-{kebab-case-title}.md
```

例:
```
adr-0001-use-jwt-for-authentication.md
adr-0002-adopt-zustand-for-state-management.md
```

## ADR の配置場所

プロジェクトルートの `docs/adr/` または `docs/decisions/` に格納することを推奨する。
