# Review: DB マイグレーション

DBマイグレーションファイルをレビューするための観点集。スキーマ変更の安全性・後方互換性・パフォーマンスを中心に確認する。

## レビュー観点

### 1. 後方互換性

| チェック項目 | OK | NG |
|------------|----|----|
| カラム削除 | アプリコードが先にカラムを参照しなくなった後に削除している | アプリが参照中のカラムを削除している |
| テーブル削除 | アプリコードが先にテーブルを参照しなくなった後に削除している | アプリが参照中のテーブルを削除している |
| カラム名変更 | 新旧カラムを並存させ、段階的に移行している | 単一マイグレーションでリネームしている |
| 型変更 | 後方互換な型変換（例: `varchar(50)` → `varchar(100)`）である | データが切り詰められる変換（例: `varchar(100)` → `varchar(50)`）である |
| NOT NULL 追加 | デフォルト値あり、または事前にNULLを除去済み | 既存データにNULLが残っている状態でNOT NULL制約を追加している |

### 2. ゼロダウンタイム運用

- スキーマ変更がデプロイ中（旧バージョンのアプリが動作中）でも安全か
- **テーブルロックを取得する操作**が含まれていないか
  - 大規模テーブルへの `ADD COLUMN`（MySQLの場合）
  - `ADD COLUMN NOT NULL DEFAULT`（PostgreSQL 10以前）
  - フルテーブルの `UPDATE` を伴うデータマイグレーション
- 大量データ更新は一括ではなくバッチ処理になっているか

### 3. インデックス設計

- **外部キーとして機能するカラムが複数ある場合、ユニーク制約の要否を確認**
  - `_id` で終わるカラムが複数存在する → 業務上の一意性制約が必要か検討
  - 既存の複合インデックスで代替できているか
- インデックス追加は `CONCURRENTLY`（PostgreSQL）や `ALGORITHM=INPLACE`（MySQL）を使っているか
- 不要なインデックスが追加されていないか（書き込みコストが増加する）
- 読み取りパフォーマンスを考慮した複合インデックスの列順になっているか

### 4. データマイグレーションの安全性

- マイグレーションと同一トランザクションで大量データを変更していないか
- バッチ処理の場合、バッチサイズと失敗時のリトライが考慮されているか
- 外部キー制約を一時的に無効化している場合、再有効化が忘れられていないか

### 5. ロールバック方針

- `up`（適用）に対応する `down`（ロールバック）が定義されているか
- データ削除や型変更など、ダウンマイグレーションが不可逆でないか確認する
- 不可逆な場合、ロールバック手順をコメントまたは migration ファイルに明記しているか

### 6. 命名規則と可読性

- テーブル名・カラム名はドメイン言語（ユビキタス言語）に沿っているか
- マイグレーションファイル名はタイムスタンプ + 変更内容が分かる名前になっているか
- コメントで変更の意図が説明されているか（特に複雑な変更の場合）

## レビューフォーマット

```
【確認した変更】
（追加・変更・削除されるテーブル/カラム/インデックスの一覧）

【問題点】
1. （問題点の説明）
   - リスク: 〇〇の可能性がある
   - 対応: △△に変更することを推奨

【懸念点】
1. （必須ではないが確認が望ましい点）

【良い点】
- 〇〇の点は適切に実装されている
```

## 合否判定の目安

**MUST FIX（マージ前に修正必須）**
- 後方互換性を壊す変更
- テーブルロックを引き起こす可能性がある操作
- NOT NULL 制約を既存データにNULLがある状態で追加

**SHOULD FIX（強く推奨）**
- ダウンマイグレーションの未定義
- 大量データ更新の一括実行
- 外部キーカラムが複数あるがユニーク制約がない

**CONSIDER（検討推奨）**
- インデックスの過不足
- 命名の一貫性
